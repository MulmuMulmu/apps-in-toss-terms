# Recapture Guidelines

## 목적

이 문서는 OCR 파이프라인이 자동 추출을 강행하지 말고 사용자에게 재촬영을 유도해야 하는 조건을 정의한다.

핵심 원칙:

- 품질이 낮은 이미지를 후처리로 무리하게 복구하려고 하지 않는다.
- 입력 품질 문제는 가능한 한 업로드 시점에 걸러낸다.
- 재촬영 기준은 사용자에게 설명 가능한 규칙이어야 한다.

## 재촬영 권장 상황

### 1. 텍스트 판독 불가

- OCR 품질 점수가 매우 낮다.
- 텍스트가 전반적으로 흐리다.
- 숫자와 한글이 다수 깨져서 날짜/품목/합계 후보가 거의 형성되지 않는다.

대표 신호:

- `quality_score` 낮음
- `low_quality_reasons`에 `blurry`, `low_quality` 포함
- `ocr_texts` 대부분이 짧은 잡음 토큰

### 2. 핵심 구간 누락

- 날짜 후보가 전혀 검출되지 않는다.
- 품목 후보가 거의 없다.
- 합계 또는 결제 금액 후보가 없다.
- 상단/하단이 잘린 것으로 보인다.

대표 신호:

- `review_reasons`에 `missing_purchased_at`
- `food_count == 0`
- `totals`가 거의 비어 있음

### 3. 품목 구간 품질 저하

- 품목명이 대부분 `unknown_item`으로 남는다.
- 수량/단위가 비어 있는 품목이 지나치게 많다.
- 잡음 줄이 품목 후보로 과도하게 들어온다.

대표 신호:

- `diagnostics.unresolved_groups` 높음
- 다수 item에 `missing_quantity_or_unit`
- 다수 item에 `low_confidence`

### 4. 구조적 손상

- 원근 왜곡이 심해 행 정렬이 깨진다.
- 그림자/반사로 특정 구간이 날아간다.
- 영수증이 배경에 묻혀 경계가 불명확하다.

## 서버/클라이언트 처리 기준

### 즉시 재촬영 권장

아래 조건이면 결과를 신뢰 가능한 데이터로 보지 않는다.

- `food_count == 0` 이고 날짜/합계도 비어 있음
- `quality_score < 0.25`
- `low_quality_reasons`에 2개 이상 존재
- 업로드 이미지에서 상단/하단 핵심 구간이 잘린 것으로 보이는 경우

### 결과는 주되, 검토 필요 표시

아래 조건이면 재촬영 강제 대신 검토 플래그를 유지한다.

- 날짜는 있으나 일부 품목만 저신뢰
- 합계/품목 일부 불일치
- 특정 품목만 `unknown_item`

이 경우:

- `review_required = true`
- `review_reasons` 반환
- 필요 시 저신뢰 품목만 Qwen 보정

## 사용자 안내 문구 예시

### 재촬영 권장

- 영수증 전체가 보이도록 다시 촬영해 주세요.
- 날짜와 합계가 잘 보이도록 상단과 하단을 포함해 주세요.
- 빛 반사나 그림자 없이 정면에서 다시 촬영해 주세요.
- 글자가 흐려서 인식이 어렵습니다. 가까이에서 다시 촬영해 주세요.

### 검토 필요

- 일부 품목만 정확하지 않을 수 있습니다.
- 추출 결과를 확인한 뒤 저장해 주세요.

## UX 원칙

- 재촬영 기준은 보수적으로 둔다.
- 사용자가 결과를 전혀 못 얻는 상황을 최소화한다.
- 다만 품질이 지나치게 낮으면 잘못된 구조화 결과보다 재촬영 유도가 낫다.
